# 一、产品概述 (Product Overview)

## 1.1 一句话定位

**面向注册性临床试验的端到端项目运营管理平台 (PMS/CTMS)**，以"项目执行 + 受试者旅程 + 数据集成中台"三轴驱动，连接申办方、研究机构、研究者、CRC、CRA、PM 及受试者/患者，实现从研究启动到锁库归档的全链路数字化运营，并在关键节点嵌入 AI 辅助（OCR 结构化、入组预测、风险预警、自动摘要），同时确保所有 AI 推理结果必须经由人类审核确认方可生效。

## 1.2 用户角色

| 角色 | 英文 | 简要说明 |
|------|------|----------|
| 管理员 | Admin | 平台超级管理员，负责系统配置、权限管理、租户/机构管理、全局字典维护 |
| 项目经理 | PM (Project Manager) | 负责单个或多个临床研究的整体运营管理：进度、质量、预算、风险、团队协调 |
| 临床监查员 | CRA (Clinical Research Associate) | 负责指定中心的监查访问（SIV/IMV/COV）、SDV/SDR、Query 跟进、中心沟通 |
| 临床研究协调员 | CRC (Clinical Research Coordinator) | 驻院/派驻研究机构，负责受试者筛选入组、访视安排、数据录入、样本管理、研究者沟通 |
| 主要研究者 | PI (Principal Investigator) | 各中心的负责人，对受试者安全与数据质量承担最终责任，审批关键决策 |
| 申办方代表 | Sponsor | 出资/发起研究的企业方人员，负责预算审批、里程碑确认、关键指标查看 |
| 财务人员 | Finance | 负责中心付款、患者补贴发放、开票与对账管理 |
| 受试者/患者 | Patient / Subject | 参与临床试验的患者，通过微信小程序完成报告上传、填写问卷、查看访视日历、在线咨询 |
| 监护人/照护者 | Caregiver | 代理受试者完成部分操作（如知情同意签署、报告上传、问卷填写） |
| 仅读审计员 | ReadOnlyAuditor | 拥有全局只读权限，用于远程稽查、监管审查、独立数据监查委员会 (DMC) 成员 |

## 1.3 关键业务目标

### 1.3.1 业务问题定义

当前临床研究项目运营中的核心痛点：

| 痛点 | 描述 | 本平台解决方案 |
|------|------|----------------|
| **信息孤岛** | PM、CRA、CRC、中心、申办方之间的信息不对称，依赖邮件/微信/Excel 传递 | 统一数据平台，角色化工作台，实时同步与通知 |
| **进度不可视** | 入组进度、访视完成率、数据录入率靠人工汇总报表，滞后严重 | 实时仪表盘、里程碑跟踪、入组预测模型 |
| **质量后知后觉** | Query、PD、AE 等质量问题只能在监查或稽查时发现 | 嵌入式质量规则引擎、实时风险热力图、自动化 Query 生成 |
| **文档管理混乱** | TMF/eTMF 文档散落在各中心、各角色手中，版本混乱、缺失风险高 | 结构化文档中心、版本控制、缺失项自动检测、归档处理 |
| **受试者脱落率高** | 受试者对访视安排、用药要求、问卷填写缺乏便捷工具，依从性差 | 微信小程序提供访视日历、提醒、ePRO、在线咨询等便捷工具 |
| **财务对账繁琐** | 中心付款、患者补贴、发票对账依赖人工核对，周期长、易出错 | 自动化对账规则、进度可视化、批量付款审批流 |
| **合规与稽查压力** | 监管稽查时需快速提供完整的审计轨迹和证据链 | 全量审计日志、基于属性的敏感字段访问控制、一键稽查报告 |
| **数据录入负担** | CRC 需手动将 HIS/EMR 数据转录至 EDC，效率低、易出错 | AI OCR 结构化 + 人工确认 + 接口对接减少转录工作量 |

### 1.3.2 核心业务价值主张

1. **让 PM 每天 15 分钟内掌握全项目真实进度** — 实时指标看板替代人工汇总周报
2. **让 CRA/CRC 从"搬运数据"变为"验证数据"** — AI OCR 结构化 + 接口集成减少 70% 手动录入
3. **让受试者"不遗忘、不脱落"** — 微信小程序全程陪伴，智能提醒 + 便捷报告
4. **让申办方"随时可看、随时可查"** — 角色化门户，里程碑与风险一览
5. **让稽查"一键追溯"** — 全链路审计日志 + 基于属性的访问控制

## 1.5 非目标范围 (Non-Goals for Phase 1)

以下内容**明确不在 Phase 1 范围**内：

| 非目标 | 说明 |
|--------|------|
| **完整 EDC (电子数据采集系统)** | Phase 1 仅做 EDC 接口适配（接收与推送数据），不构建独立的 eCRF 设计器、Edit Check 编程等 EDC 核心功能 |
| **完整 eTMF (电子试验主文件)** | Phase 1 仅做文档中心的分类/版本/上传/审批，不做 eTMF reference model 的完整 DIA/TMF RM 映射 |
| **随机化系统 (IRT/IWRS)** | Phase 1 仅支持手动录入随机号/药物编号与简单的区组随机逻辑配置，不做完整的供应链管理、药物盲法库存管理 |
| **药物供应链管理** | 不在 Phase 1 范围 |
| **独立统计编程环境** | 数据分析仅做导出，不做 SAS/R 编程集成 |
| **移动端 CRF 录入 App (CRA 用)** | 管理端为 Web 响应式，暂不独立开发 CRA/CRC 移动端 App |
| **语音/视频问诊集成** | 在线咨询以图文为主，视频/语音为 Phase 2 |
| **多语言国际化** | Phase 1 仅中文（简体），但信息架构预留多语言扩展点 |
| **SSO/LDAP 集成** | Phase 1 使用平台本地认证，SSO 为 Phase 2 |
| **BI/OLAP 分析引擎** | Phase 1 报表由 OpenSearch + 预计算实现，不做独立 BI 引擎 |
| **自动化测试全覆盖** | Phase 1 核心流程有集成测试，非真 100% 覆盖 |

## 1.6 关键假设 (Key Assumptions)

| 序号 | 假设 | 风险等级 | 备注 |
|------|------|----------|------|
| A1 | 研究机构（医院）有基本的 HIS/LIS/PACS 系统，且能提供数据导出或接口对接合作 | 中 | 无接口时降级为 OCR 上传模式 |
| A2 | 研究机构允许 CRC/研究者使用院内网络访问本平台 | 低 | 前提：通过等保认证与医院信息安全审核 |
| A3 | 受试者使用微信且愿意安装小程序 | 低 | 微信覆盖率在中国极高 |
| A4 | 申办方使用主流财务系统（用友/金蝶/SAP）并可导出对账文件 | 中 | 对账模块需支持文件导入作为备选方案 |
| A5 | CRA 监查访问频率、监查报告模板等遵循 GCP 与申办方 SOP | 低 | 可通过系统配置适配不同申办方 SOP |
| A6 | 每个研究项目的科室、访视方案、问卷量表等由项目初始化时配置完毕 | 低 | 提供配置模板库减少重复工作 |
| A7 | Phase 1 接入 ≤3 个中心（单项目），系统架构在 Phase 2 扩展至多中心 | 低 | 软件架构从 Day 1 即支持多中心设计 |
| A8 | 实时数据同步（如 HIS 对接）非 Phase 1 强制需求，T+1 批处理可接受 | 低 | 架构预留实时通道 |
| A9 | AI OCR 服务的识别准确率需达到 95%+ 方可直接使用，否则需人工修正 | 低 | 所有 OCR 结果均需人类确认 |
| A10 | Flowable 工作流用于审批流（如预算审批、CAPA 签核）而非核心业务状态机 | 低 | 核心业务状态由应用代码 + 数据库事务管理 |

## 1.7 核心用户画像与旅程 (User Personas & Core Journeys)

```
每日开始 (09:00)
  │
  ├──> 打开工作台 → 查看项目整体入组仪表盘 → 确认昨日入组数据无异常
  ├──> 审查风险热力图 → 标记高风险中心 → 分配给 CRA 跟进
  ├──> 处理待办列表 → 审批 MVR / 审批预算变更 / 回复升级 Query
  │
每日中段 (10:00 – 16:00)
  ├──> 审核监查报告 (MVR) → 批注 → 批准
  ├──> 参加项目团队会议 → 查看自动生成的周报摘要草案
  ├──> 与申办方沟通里程碑进展 → 导出项目进展报告（脱敏版）
  ├──> 审查新发现的 Protocol Deviation → 判断是否需要 CAPA
  │
每日结束 (17:00)
  ├──> 查看今日自动摘要：关键指标变化/新增问题/超期待办
  ├──> 确认明日重点：到期访视/到期 Query/到期里程碑
```

```
  │     ├── 审查该中心近期 Query 状态
  │     ├── 审查该中心近期 PD / AE 记录

  ├──> 到达中心 → 系统记录到达时间
```

## 1.8 系统核心设计原则补充

### 1.8.1 数据一致性保证策略

| 场景 | 一致性级别 | 实现方式 |
|------|-----------|----------|
| 受试者入组 + 访视计划生成 | 强一致 (事务) | Spring @Transactional，同一数据库事务 |
| 访视完成 + 补贴计算触发 | 最终一致 (事件) | 发布事件 → RabbitMQ → 消费计算 |
| OCR 结果确认 + 数据同步 | 最终一致 (异步) | AI Service → 回调 → 状态更新 |
| AE 创建 + 安全通知 | 最终一致 (事件) | 应用事务提交后发布事件 |
| 导出审批 + 文件生成 | 最终一致 (异步) | 审批通过 → 后台任务 → 通知下载 |

### 1.8.2 AI 服务调用安全边界

```
┌─────────────────────────────────────────────────────────────────┐
│  AI/OCR Service (独立部署)                                       │
│                                                                  │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  允许的操作:                                               │  │
│  │  - 读取: 从 MinIO 读取上传的图片/文档                      │  │
│  │  - 写入: OCR 结构化结果写入 PostgreSQL (ai_results schema) │  │
│  │  - 通知: 回调主后端 /api/internal/ai-callback (Service-to-│  │
│  │          Service Auth)                                     │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                  │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  禁止的操作:                                               │  │
│  │  - 直接修改核心业务表 (subjects, visits, aes 等)           │  │
│  │  - 直接发起审批/状态流转                                    │  │
│  │  - 直接发送用户通知 (需通过主后端)                          │  │
│  │  - 绕过审计日志写入                                         │  │
│  └───────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

### 1.8.3 审批流与状态机的职责边界

- **业务状态机**：由应用代码在数据库事务中管理，确保强一致性。例如受试者"筛选 → 入组"状态变更。
- **审批流**：由 Flowable 工作流引擎管理，处理需要多人协作的审批场景。例如预算审批需要 PM → 申办方 → 财务三级审批。
- **关键设计约束**：审批流的结果可以触发状态机变更（如预算审批通过后将预算状态改为"已批准"），但状态机本身不受 Flowable 直接控制，而是由应用代码在收到审批结果回调后执行。

## 2.3 管理端与患者端关键跳转关系

| 触发场景 | 发送端 | 接收端 | 跳转方式 |
|----------|--------|--------|----------|
| CRA/CRC 发现 Query 需要受试者确认 | 管理端 Query 详情 → "发送给受试者确认" | 患者端消息 → Query 确认页 | 微信服务通知 + 模板消息 |
| 系统推送访视提醒 | 管理端访视计划 → 自动触发 | 患者端消息 → 访视详情页 | 微信服务通知 + 小程序内跳转 |
| 受试者上传报告 OCR 完成 | AI 服务 → 回调通知 | 患者端消息 → 报告确认页 | 微信服务通知 |
| 受试者自报 AE | 患者端 → 填写 AE 自报表单 | 管理端安全模块 → 新 AE 通知 | 管理端消息 + 邮件通知 |
| CRC/PM 发送在线咨询回复 | 管理端消息 → 回复 | 患者端消息 → 对话详情 | 微信服务通知 |
| 申办方审批预算通过 | 管理端工作流 → 审批通过通知 | PM 管理端 → 财务中心 | 管理端站内通知 |
| 受试者 eConsent 签署完成 | 患者端 → 签章完成 | 管理端文档中心 → 知情同意书归档 | 系统自动归档 + 通知 |
| 受试者提交报销申请 | 患者端 → 填写报销表单 | 管理端财务 → 报销审批 | 微信服务通知 + 站内通知 |
| PM 发布研究公告 | 管理端消息 → 发布公告 | 患者端首页 → 公告卡片 | 微信服务通知 + 首页展示 |
| 系统检测到访视窗口超期 | 自动规则引擎触发 | PM/CRA/CRC 管理端 → 待办 | 站内通知 + 邮件 |

### 模块 M01：项目组合与工作台 (Portfolio & Workspace)

#### 功能边界

**In Scope：**
- 项目组合仪表盘：多项目聚合视图，展示所有项目的核心指标卡片（入组率/完成率/Query 统计/安全事件统计）
- 项目列表：支持搜索（项目名称/申办方/治疗领域/适应症/状态）、筛选、排序、分页
- 项目生命周期管理：创建研究项目（初始化方案、中心、预算等必要信息）、状态流转（创建→启动中→入组中→入组完成→数据清理→锁库→关闭）、项目归档
- 角色化工作台：根据当前登录用户角色自动加载对应的首页视图（详见 2.4 节）
- 待办事项聚合：跨模块拉取当前用户的待办任务（审批、签字、回复 Query 等）统一展示在工作台
- 个人快捷入口：用户可自定义常用功能入口
- 公告管理：PM/Admin 可发布项目级或系统级公告，指定可见角色范围

**Out of Scope (Phase 1)：**
- 跨申办方的多租户 SaaS 隔离（Phase 1 以平台模式运行，不区分租户）
- 高级项目组合分析（项目组合 OPTIMIZATION、资源平衡等）
- 项目模板一键复制（Phase 1 需手动创建项目）

#### 模块间关系

| 关联模块 | 关系说明 |
|----------|----------|
| M02 研究启动与立项 | 工作台提供"创建项目"入口，跳转至 M02 创建启动流程 |
| M07 监查管理 | 工作台聚合监查待办（待提交 MVR、待审批 MVR） |
| M08 质量管理 | 工作台聚合质量待办（待回复 Query、待处理 PD、待执行 CAPA） |

#### 关键状态机

**项目生命周期状态机：**

```
[创建] ──(初始化完成)──> [启动中] ──(首个中心通过伦理)──> [入组中]
  ^                                                          │
  │                                                          ▼
  └────────────────────(重新激活)──────────── [入组完成] ──(数据清理开始)──> [数据清理中]
                                                │                              │
                                                ▼                              ▼
                                           [暂停/Suspended]              [锁库/Locked]
                                                │                              │
                                                └──(恢复)──> [入组中]          ▼
                                                                          [关闭/Closed]
                                                                              │
                                                                              ▼
                                                                         [归档/Archived]
```

状态转换条件：
- **创建 → 启动中**：PM 完成项目基本信息填写、关联方案、配置初始预算
- **启动中 → 入组中**：至少一个中心通过伦理委员会批准
- **入组中 → 入组完成**：实际入组数 ≥ 计划入组数
- **入组完成 → 数据清理中**：最后一个受试者末次访视完成后
- **数据清理中 → 锁库**：所有 Query 已关闭、所有数据已确认
- **任何阶段 → 暂停**：仅 Admin/Sponsor 可操作（如安全性原因）
- **关闭 → 归档**：所有中心已关闭、所有文档已归档

### 模块 M06：访视计划与访视执行 (Visit Planning & Execution)

#### 功能边界

**In Scope：**
- 访视计划模板：基于方案访视 Schedule 创建标准访视计划模板（访视名称、访视序号、访视窗 (±天数)、检查项目清单、样本采集清单、问卷清单、数据录入清单）
- 受试者访视计划生成：受试者入组后，基于入组日期 + 访视计划模板自动生成该受试者的全部计划访视
- 访视日历视图：以日历形式展示所有受试者的计划访视与已完成访视，支持按中心/状态筛选
- 访视执行管理：访视签到（计划访视/非计划访视）、检查项目完成状态勾选、样本采集记录（样本类型/采集时间/条码号/送检状态）、数据录入状态跟踪
- 访视窗管理：自动标注访视窗状态（即将到达/在窗/已超窗）、超窗提醒与升级通知
- 非计划访视 (Unscheduled Visit)：支持记录计划外访视，关联原因（AE 随访/重复检查等）
- 访视数据完整性校验：访视完成的必要条件校验（必做检查完成、必填数据完成）
- 电话访视支持：标记为电话访视，记录通话摘要

**Out of Scope (Phase 1)：**
- 视频访视功能（由在线咨询功能间接支持）
- 访视排程自动化排期算法（需配合 CRA/CRC 可用时间表，Phase 1 手动排程）
- DCT (Decentralized Clinical Trial) 上门访视调度

#### 模块间关系

| 关联模块 | 关系说明 |

| M03 方案管理 | 访视计划模板的数据来源（访视 Schedule + 检查项目） |

**访视状态机：**

```
[计划/Scheduled] ──(窗口期开始)──> [待执行/Pending]
                                      │
                          ┌───────────┼───────────┐
                          ▼           ▼           ▼
                    [已完成/Completed] [已超窗/Missed] [跳过/Skipped]
```

**访视窗状态计算：**
- 目标访视日期 T0 = 入组日期 + 方案规定的访视间隔天数

- 窗结束 = T0 + 窗上限天数

### 模块 M08：Query/Issue/Protocol Deviation/CAPA

#### 功能边界

**In Scope：**

**Query 管理：**
- Query 创建：手动创建（CRA/PM/数据管理员 发现数据问题）、自动创建（质量规则引擎触发，如数据范围异常、缺失、逻辑矛盾）
- Query 分类：按严重性 (Critical/Major/Minor)、按类别（数据录入错误/数据缺失/数据矛盾/逻辑不一致/源数据匹配/其他）
- Query 生命周期：创建 → 开启 → 分配 → 回复 → 审核 → 关闭 / 重新开启
- Query 对话线：每个 Query 支持多轮回复，形成完整的问题处理对话记录
- Query 统计：按中心/按项目统计 Query 数量、类型分布、平均关闭时间、回复质量评分

**Protocol Deviation (PD) 管理：**
- PD 记录：PD 类型（入组标准偏离/访视超窗/用药错误/检查漏做/样本处理不当/知情同意问题/其他）、PD 严重性（重大/Minor）、PD 发现来源（CRA 监查/CRC 自我报告/数据核查）
- PD 判定流程：CRC/CRA 记录 → PI 判定（是否构成 PD/严重性分级）→ PM 确认
- PD 关联 CAPA：关键 PD 可关联至 CAPA 流程
- PD 统计：按中心/类型/阶段统计 PD 发生率与趋势

**CAPA 管理：**
- CAPA 创建：来源可为 Query 趋势分析、PD 分析、稽查发现、质控发现
- CAPA 流程：问题描述 → 根因分析 (Root Cause Analysis) → 纠正措施 (Corrective Action) → 预防措施 (Preventive Action) → 实施计划 → 实施跟踪 → 效果验证 → 关闭
- CAPA 审批（Flowable）：创建 → PM 审批 → 申办方确认（重大 CAPA）→ 执行 → 效果验证 → 关闭
- CAPA 跟踪：责任人/截止日期/完成进度/效果验证指标

**Out of Scope (Phase 1)：**
- AI 自动分析 Query 趋势并建议 CAPA
- 医学编码自动匹配 (MedDRA/WHO Drug)（Phase 2 考虑集成编码工具）
- 正式 Data Cleaning Plan (DCP) 管理
- 外部稽查发现管理（可作为特殊类型 CAPA 记录）

### 模块 M09：AE/SAE/安全上报与升级 (Safety Reporting & Escalation)

#### 功能边界

**In Scope：**
- AE (不良事件) 记录：AE 名称（MedDRA 编码预留）、AE 严重等级 (CTCAE Grade 1-5)、与研究药物的关系评估（无关/可能无关/可能有关/很可能有关/肯定有关）、AE 发生日期/结束日期、处理措施、转归
- SAE (严重不良事件) 记录：SAE 标准判断（死亡/危及生命/住院或延长住院/导致显著残疾/致畸/其他重要医学事件）、SAE 上报时限跟踪（24小时内报告申办方、7/15天内报告监管机构）、SUSAR（非预期严重不良反应）判断与上报跟踪
- 安全上报流程（Flowable）：CRC 初录 AE → PI 评估（严重性 + 关联性）→ 上报申办方 → 申办方安全部门评估 → 必要时上报药监部门
- 安全事件升级：SAE/SUSAR 自动升级通知给 PM/Sponsor/安全部门、超时未上报升级通知
- 安全数据汇总：AE 汇总表（按 SOC/PT 分类）、SAE 列表、安全趋势分析、按中心/项目统计 AE 发生率
- 合并用药记录：药品名称、剂量、频次、起止日期、适应症、是否方案允许

**Out of Scope (Phase 1)：**
- MedDRA Browser 在线编码工具集成（提供 MedDRA 编码字段手工输入）
- E2B(R3) XML 格式生成与电子上报（提供数据导出供 SAS 程序员/安全部门处理）
- DSUR (Development Safety Update Report) 自动生成（仅提供所需数据导出）
- 独立 Data Safety Monitoring Board (DSMB) 门户

| M12 消息协作 | 安全升级通知推送、上报流程中的审批通知 |
| M10 文档中心 | SAE 报告、安全评估报告归档 |
| P08 AE 自报 | 受试者通过小程序自报的不适症状自动创建为 AE 草稿记录 |

```
[上报监管/Reported to HA] (Phase 1 手工跟踪)  [关闭/Closed]
```

**CTCAE 严重等级参考：**
- Grade 1: 轻度 (Mild)
- Grade 2: 中度 (Moderate)
- Grade 3: 重度 (Severe)
- Grade 4: 威胁生命 (Life-threatening)
- Grade 5: 死亡 (Death related to AE)

**关联性判断：**
- Not Related (无关)
- Unlikely Related (可能无关)
- Possibly Related (可能有关)
- Probably Related (很可能有关)
- Definitely Related (肯定有关)

### 模块 H02：HIS/LIS/PACS/EMR 接口适配 (Hospital System Adapters)

#### 功能边界

**In Scope：**
- 标准化接口适配层：定义统一的院内系统连接接口规范
- HL7 v2 / FHIR R4 基础支持：支持 HL7 v2.x 消息解析（ADT/ORM/ORU）、支持 FHIR R4 基础资源 (Patient/Observation/DiagnosticReport 等)

### 模块 H04：文档与附件分层存储 (Document & Attachment Tiered Storage)

#### 功能边界

**In Scope：**
- 三层存储架构：
  - 热数据层：最近 90 天内活跃的文档（MinIO on SSD / 本地存储）
  - 温数据层：90 天至 1 年的文档（MinIO on HDD）

- 版本化存储：每次文档更新存储为新的对象版本

### 模块 H06：数据质量规则引擎 (Data Quality Rule Engine)

#### 功能边界

**In Scope：**
- 规则类型：
  - 范围检查：数值超出正常/参考范围
  - 缺失检查：必填数据项为空
  - 逻辑检查：数据与数据的逻辑矛盾（如 AE 结束日期早于开始日期）
  - 访视窗检查：访视是否在允许的窗口内完成
  - 异常变化检查：连续两次访视的数据变化超过阈值（如体重变化超过 10%）
  - 依从性检查：eDiary 连续漏填天数超过阈值
- 规则配置管理：规则名称、规则类型、作用域（项目级/中心级）、数据对象（表单/字段）、触发条件、严重级别（Error/Warning/Info）、自动动作（自动创建 Query / 仅标记 / 通知）
- 规则执行：事件驱动（消费事件总线消息）+ 定时批量检查（T+1）、执行日志与结果追踪
- AI 辅助规则建议：基于历史数据异常模式的质量规则建议（仅供人工审核）

**Out of Scope (Phase 1)：**
- 复杂的统计异常检测 (Statistical Outlier Detection)
- 完整的 Edit Check 编程环境（类似 EDC 中的 Edit Check Specification）
- 医学逻辑一致性检查 (Medical Logic Modules, Arden Syntax)

### 模块 H10：脱敏与权限控制 (Masking & Access Control)

#### 功能边界

**In Scope：**
- 数据分级策略：

| 敏感级别 | 字段示例 | 默认脱敏策略 | 可见角色 |
|----------|---------|-------------|----------|
| P0 (公开) | 受试者编号、性别、出生年份 | 无脱敏 | 所有授权用户 |
| P1 (内部) | AE 详情、合并用药、检查结果 | 无脱敏 | PM/CRA/CRC/PI/Sponsor |
| P2 (敏感) | 受试者全名 | 首字母 + ** (如 张**) | 仅直属 CRC、直属 PI |
| P3 (高度敏感) | 身份证号、电话号码、银行卡号 | 前3后4 (如 310****1234) | 仅直属 CRC (有业务需要时临时查看) |
| P4 (极度敏感) | 基因数据、HIV 状态 | 完全不显示/不可导出 (Phase 1) | 仅 PI + Sponsor 安全部门 |

- 动态脱敏 (Dynamic Data Masking)：API 层根据请求用户角色 + 数据所有权 + 敏感级别自动应用脱敏规则
- 静态脱敏 (Static Data Masking)：导出/分析等场景下的离线数据脱敏
- 脱敏规则配置：敏感字段定义（数据库字段级）、脱敏算法选择（截断/替换/哈希/泛化）
- 临时脱敏解除：受试者全名/联系电话 等字段在特定业务场景（如 CRC 联系受试者）可临时查看完整信息、查看操作需记录审计日志

**Out of Scope (Phase 1)：**
- 数据分类分级自动化扫描
- 同态加密 (Homomorphic Encryption)
- 隐私计算/联邦学习

#### 模块间关系

| 关联模块 | 关系说明 |
|----------|----------|
| M14 系统配置 | 脱敏规则与权限配置 |
| 所有业务模块 | 数据读取时动态调用脱敏服务 |
| H07 数据导入导出 | 导出文件的静态脱敏 |

| Query 即将超期 | 站内通知 | 否 | 是 | 到期前 24h |

| 审批待办 (PM/PI/Sponsor) | 工作台待办 | 否 | 是 | 实时 |

| 策略 | 说明 |
|------|------|

## 附录 F：Phase 1 数据保留与合规策略

### F.1 数据保留周期

| 数据类型 | 在线保留 (OpenSearch) | 归档保留 (冷存储) | 备注 |
|----------|----------------------|-------------------|------|
| 审计日志 | 3 年 | 5 年以上 | 依 GCP 要求需保存至试验结束后至少 5 年 |
| 受试者文档 | 研究关闭后 2 年 | 15 年 | 依药品注册管理办法 |
| AE/SAE 记录 | 永久 (在线) | — | 安全性数据需永久保留 |
| Query/PD/CAPA | 研究关闭后 2 年 | 15 年 | 与 TMF 归档周期一致 |
| 财务记录 | 研究关闭后 2 年 | 10 年 | 依财务/税务法规 |
| 运行日志 | 30 天 | 1 年 | 运维诊断用途 |
| OCR 原始图片 | 3 个月 (OCR确认后) | 研究关闭 | 已确认数据可清理原始图片 |

### F.2 数据销毁流程

1. 系统管理员发起数据销毁申请（需注明销毁范围 + 合规依据）
2. Sponsor 合规部门审批
3. 检索确认：确认销毁范围不包括法规要求的保留数据
4. 执行销毁：数据库记录 + 对象存储文件 + 搜索引擎索引
5. 生成销毁证明：销毁时间/范围/执行人/审批链 记录为不可篡改的审计日志
